"그냥 짜 줘"라고 말하면 가장 빠를 것 같습니다.

하지만 실제로는 자주 느립니다.
처음 코드는 빨리 나오지만, 곧바로 이런 일이 생깁니다.

"이건 내가 원한 흐름이 아닌데?"
"왜 기존 기능이 깨졌지?"
"수정했더니 다른 화면이 이상해졌네?"

문제는 AI가 코드를 못 짜서가 아닙니다.
대부분은 **내 머릿속에 있는 목표가 AI에게 충분히 전달되지 않았기 때문**입니다.

AI 코딩 요청의 핵심은 어렵지 않습니다.

**내가 만들고 싶은 흐름을 AI와 함께 구체화하고, 그걸 작은 작업 단위로 쪼갠 뒤, 각 단계마다 계획물·산출물·검증 결과를 받는 것.**

이게 전부입니다.

사용자가 모든 파일 구조, 상태 관리, 테스트 방법을 직접 알아야 한다는 뜻이 아닙니다.
오히려 그런 기술적인 분석은 AI에게 시키면 됩니다.
대신 사용자는 AI가 방향을 잘 잡을 수 있도록 아래 세 가지를 분명히 말해야 합니다.

1. 내가 만들고 싶은 사용자 흐름
2. 이번 작업에서 중요한 기준
3. 한 번에 다 하지 말고 작게 나눠서 진행하라는 방식

이 글은 그걸 실제로 어떻게 말하면 되는지에 대한 글입니다.

## 핵심 요약

AI에게 코딩을 맡길 때는 이렇게 생각하면 됩니다.

1. **머릿속 내용을 말로 꺼낸다**
2. **AI에게 질문하게 해서 요구사항을 구체화한다**
3. **AI에게 작업을 아주 작은 단계로 나누게 한다**
4. **한 단계씩 구현하고 검증 결과를 받는다**
5. **막히면 바로 고치지 말고 원인부터 찾게 한다**

복잡한 말로 하면 계획, 산출물, 검증입니다.
간단히 말하면 이겁니다.

**"내가 원하는 걸 같이 정리하고, 작게 나눠서, 하나씩 제대로 만들어줘."**

## 프롬프트는 예시처럼 완벽할 필요가 없다

이 글에 나오는 프롬프트 예시는 일부러 온전하게 적어둔 것입니다.
그대로 복사해서 써도 되지만, 매번 저렇게 길게 쓸 필요는 없습니다.

중요한 건 문장이 아니라 핵심 조건입니다.

AI에게 전달해야 할 핵심은 아래 정도입니다.

1. 내가 원하는 사용자 흐름
2. 아직 모르는 부분은 AI가 질문해야 한다는 것
3. 바로 구현하지 말고 먼저 계획해야 한다는 것
4. 작업을 작은 단위로 쪼개야 한다는 것
5. 각 단계마다 산출물과 검증 방법이 있어야 한다는 것
6. 단계 밖의 작업은 하지 말아야 한다는 것

이 여섯 가지만 지키면 프롬프트는 짧아도 됩니다.

예를 들어 처음 요청은 이 정도로 줄여도 됩니다.

```text
내가 원하는 흐름은 [원하는 사용자 흐름]이야.

부족한 요구사항은 먼저 질문해줘.
바로 구현하지 말고 현재 코드 구조를 보고 작은 단계로 계획해줘.
각 단계마다 산출물과 검증 방법을 적어줘.
```

구현할 때도 길게 말할 필요는 없습니다.

```text
방금 계획의 다음 단계만 진행해줘.

단계 밖의 작업은 하지 말고,
끝나면 변경 파일, 산출물, 검증 결과만 알려줘.
```

프롬프트를 잘 쓰는 사람은 멋진 문장을 쓰는 사람이 아닙니다.
AI가 놓치면 안 되는 기준을 반복해서 잡아주는 사람입니다.

## 1단계: 머릿속 기능을 먼저 충분히 설명하기

가장 흔한 요청은 이런 식입니다.

```text
팀 캘린더에서 가능한 회의 시간을 추천하는 기능 만들어줘.
```

틀린 요청은 아닙니다.
하지만 AI 입장에서는 모르는 게 너무 많습니다.

예를 들면 이런 것들입니다.

1. 누가 회의 참석자인지 어떻게 고르는가
2. 추천 시간은 어디에 보여줄 것인가
3. 추천 시간을 누르면 어떤 화면으로 이어지는가
4. 기존 일정과 겹치면 어떻게 처리할 것인가
5. 저장까지 해야 하는가, 초안까지만 만들면 되는가

이걸 사용자가 처음부터 완벽하게 정리할 필요는 없습니다.
대신 머릿속에 있는 흐름을 말하듯이 꺼내면 됩니다.

```text
내가 만들고 싶은 건 대략 이런 흐름이야.

팀 캘린더에서 회의 참석자를 고르면,
참석자들의 일정과 겹치지 않는 시간이 몇 개 추천되고,
추천 시간 하나를 누르면 회의 만들기 화면에 그 시간이 자동으로 들어갔으면 좋겠어.

처음부터 저장까지 완성하지 않아도 돼.
먼저 사용자가 추천 시간을 보고 선택하는 흐름이 자연스러운지 확인하고 싶어.
```

이 정도만 말해도 훨씬 좋아집니다.
파일명이나 함수명은 몰라도 됩니다.
중요한 건 **사용자가 실제로 어떤 경험을 하길 원하는지**를 AI가 이해하게 만드는 것입니다.

## 2단계: 바로 구현하지 말고 AI에게 질문하게 하기

내가 원하는 흐름을 말했으면, 바로 "구현해줘"로 넘어가지 않는 게 좋습니다.
아직 빈칸이 많기 때문입니다.

이때는 AI에게 먼저 질문하게 해야 합니다.

```text
바로 구현하지 말고,
이 기능을 제대로 만들기 위해 필요한 질문을 먼저 해줘.

내가 답해야 할 제품 흐름 질문과,
네가 코드에서 확인해야 할 기술 질문을 나눠서 정리해줘.
```

그러면 AI는 보통 이런 식으로 나눠야 합니다.

```text
사용자에게 확인할 것:
1. 추천 시간은 캘린더 위에 보여줄지, 별도 패널에 보여줄지
2. 추천 후보는 몇 개까지 보여줄지
3. 추천 시간을 누르면 바로 저장할지, 회의 초안만 만들지

코드에서 확인할 것:
1. 참석자 선택 상태가 어디서 관리되는지
2. 일정 데이터가 어떤 형태로 저장되는지
3. 회의 생성 화면에 시간을 주입하는 기존 방법이 있는지
4. 시간대 처리가 이미 구현되어 있는지
```

이 단계가 중요합니다.
사용자가 모든 질문을 직접 떠올릴 필요가 없습니다.
AI에게 "내가 빠뜨린 걸 찾아줘"라고 시키면 됩니다.

AI 코딩은 프롬프트를 길게 쓰는 게임이 아닙니다.
**내 생각을 AI와 대화하면서 빠진 부분 없이 구체화하는 과정**에 가깝습니다.

## 3단계: 작업을 작게 쪼개게 하기

요구사항이 어느 정도 정리되면, 이제 계획을 만들게 합니다.
여기서 핵심은 "잘 짜인 큰 계획"이 아닙니다.
**실패했을 때 원인을 바로 찾을 수 있을 만큼 작은 계획**입니다.

나쁜 계획은 이런 식입니다.

```text
1. 참석자 선택 기능 추가
2. 가능한 시간 계산 기능 추가
3. 추천 UI 추가
4. 저장 기능 연결
5. 테스트
```

겉으로는 괜찮아 보이지만 아직 큽니다.
예를 들어 2번에서 문제가 나면 데이터 조회 문제인지, 시간 계산 문제인지, 시간대 문제인지 바로 알기 어렵습니다.

이럴 때는 이렇게 요청하면 됩니다.

```text
이 계획은 아직 크다.
실패했을 때 원인을 바로 찾을 수 있게 더 작은 단위로 쪼개줘.

한 단계에는 하나의 변화만 들어가게 해줘.
각 단계마다 산출물과 검증 방법도 같이 적어줘.
아직 구현하지 마.
```

좋은 계획은 이런 모양에 가깝습니다.

```text
1단계: 현재 코드 구조 확인
산출물: 관련 파일, 데이터 흐름, 기존 회의 생성 흐름 요약
검증: 기존 캘린더와 회의 생성 기능이 어디서 동작하는지 확인

2단계: 참석자 선택 상태 읽기
산출물: 선택된 참석자 목록을 읽을 수 있는 연결
검증: 참석자를 바꿨을 때 선택 상태가 정상적으로 바뀌는지 확인

3단계: 일정 데이터 읽기
산출물: 선택된 참석자들의 일정 목록 확보
검증: 날짜 범위와 시간대가 기존 캘린더 기준과 맞는지 확인

4단계: 빈 시간 후보 계산
산출물: 화면과 분리된 계산 함수 또는 유틸
검증: 겹치는 일정, 빈 일정, 하루 끝 시간 같은 케이스 확인

5단계: 추천 시간 목록 표시
산출물: 저장 기능 없는 추천 목록 UI
검증: 후보가 있을 때와 없을 때 화면이 깨지지 않는지 확인

6단계: 후보 클릭 시 회의 초안에 반영
산출물: 선택한 시간이 회의 만들기 화면에 들어가는 연결
검증: 기존 회의 생성 흐름이 깨지지 않는지 확인

7단계: 최종 정리와 회귀 확인
산출물: 변경 파일 요약, 남은 위험, 테스트 결과
검증: 기존 일정 보기, 회의 생성, 추천 시간 선택 흐름 확인
```

사용자가 이걸 직접 다 적으라는 뜻이 아닙니다.
AI에게 이렇게 만들라고 시키면 됩니다.

여기서 사용자가 할 일은 세부 구현을 설계하는 게 아닙니다.
아래 정도만 판단하면 됩니다.

1. 단계가 너무 크지 않은가
2. 내가 원하는 사용자 흐름과 맞는가
3. 저장, 삭제, 결제, 전송처럼 위험한 연결이 너무 빨리 나오지 않았는가
4. 각 단계마다 확인할 결과물이 있는가

## 4단계: 한 번에 전부 구현하지 않기

계획이 마음에 든다고 바로 "전체 진행해줘"라고 하면 다시 위험해집니다.
기능이 복잡할수록 한 단계씩 가는 편이 빠릅니다.

이렇게 요청하세요.

```text
계획의 1단계만 진행해줘.

이번 단계 밖의 작업은 하지 마.
구현 후에는 변경 파일, 산출물, 검증 결과를 짧게 알려줘.
진행하다가 1단계가 생각보다 크다고 판단되면 더 쪼개고 멈춰.
```

다음 단계도 똑같습니다.

```text
2단계 진행해줘.

이전 단계에서 확인한 구조를 기준으로 진행해줘.
이번 단계 밖의 작업은 하지 마.
검증까지 끝낸 뒤 결과를 알려줘.
```

이 방식의 장점은 분명합니다.

1. 어디서 문제가 생겼는지 추적하기 쉽다
2. 원하지 않는 방향으로 크게 벗어나기 전에 멈출 수 있다
3. 기존 기능을 깨뜨릴 가능성이 줄어든다
4. 사용자가 기술 디테일을 몰라도 진행 상황을 판단할 수 있다

AI에게 일을 적게 시키는 게 아닙니다.
**한 번에 섞어서 시키지 않는 것**입니다.

## 각 단계가 끝나면 무엇을 받아야 하나

작게 나눠서 진행하는 이유는 통제하기 위해서입니다.
그러려면 각 단계가 끝날 때 결과를 그냥 "완료"로 받으면 안 됩니다.

최소한 아래 네 가지는 받아야 합니다.

1. 어떤 파일이 바뀌었는지
2. 사용자가 보게 되는 동작이 무엇이 달라졌는지
3. 어떤 방식으로 검증했는지
4. 다음 단계 전에 남은 위험이나 확인할 점이 무엇인지

예를 들면 이렇게 요청하면 됩니다.

```text
이번 단계가 끝나면 아래 형식으로 정리해줘.

1. 변경 파일
2. 바뀐 동작
3. 검증 결과
4. 다음 단계 전에 확인할 것
```

이렇게 받아두면 다음 요청이 쉬워집니다.
다음에 그냥 "다음 단계 진행해줘"라고 해도, AI가 이전 결과를 기준으로 이어갈 수 있습니다.

반대로 매 단계마다 긴 회고 문서를 만들 필요는 없습니다.
작은 수정이라면 마지막 요약만으로 충분합니다.

## 5단계: 막히면 "고쳐줘"보다 "원인부터 찾아줘"

버그가 났을 때 바로 이렇게 말하기 쉽습니다.

```text
저장 버튼이 안 돼. 고쳐줘.
```

하지만 이러면 AI가 추측으로 코드를 바꿀 수 있습니다.
그러면 다른 곳이 깨질 수 있습니다.

버그가 났을 때는 이렇게 말하는 편이 안전합니다.

```text
지금 저장 버튼을 눌러도 메모가 추가되지 않는다.

바로 수정하지 말고 원인부터 특정해줘.
확인할 후보는 네가 코드 구조를 보고 정해줘.

원인을 찾은 뒤 가장 작은 수정만 적용해줘.
수정 후에는 원인, 변경 파일, 검증 결과를 알려줘.
```

프론트엔드 문제라면 이런 식으로 덧붙이면 더 좋습니다.

```text
특히 이벤트 연결, 상태 변경, 렌더링, 저장 요청, 복원 흐름, CSS 레이어 충돌을 확인해줘.
```

사용자가 이 순서를 다 알 필요는 없습니다.
그냥 **"추측해서 고치지 말고 원인부터 찾아라"**라고 시키면 됩니다.

## 방향이 바뀌면 기존 구현에 억지로 붙이지 않기

AI로 만들다 보면 중간에 생각이 바뀝니다.
이건 실패가 아닙니다.
오히려 실제로 써보면서 더 나은 흐름을 찾은 것입니다.

예를 들어 처음에는 추천 시간을 캘린더 위에 띄우려고 했는데, 써보니 일정과 겹쳐서 보기 어렵다고 느낄 수 있습니다.
그러면 이렇게 말하면 됩니다.

```text
목표가 바뀌었다.

처음에는 추천 시간을 캘린더 위에 띄우려고 했지만,
실제로 보니 기존 일정과 겹쳐서 비교하기 어렵다.

추천 시간은 캘린더 위가 아니라 우측 패널 목록으로 보여주고 싶다.
캘린더는 기존 일정과 빈 시간을 확인하는 역할만 하게 두고,
선택은 우측 패널에서 하게 바꾸고 싶다.

기존 구현을 억지로 유지하지 말고,
새 흐름 기준으로 유지할 것과 제거할 것을 먼저 나눠줘.
아직 구현하지 마.
```

핵심은 "지금 만든 게 아까우니까 어떻게든 살려줘"가 아닙니다.
**새로 알게 된 사용자 흐름에 맞춰 구조를 다시 정리하게 하는 것**입니다.

## 바로 복사해서 쓰는 기본 템플릿

처음 요청은 이 정도면 충분합니다.

```text
내가 만들고 싶은 건 대략 이런 흐름이야.

[사용자가 실제로 경험하길 원하는 흐름을 말하듯 적기]

나는 세부 구현 순서를 잘 모른다.
바로 구현하지 말고 먼저 나와 함께 요구사항을 구체화해줘.

1. 내가 답해야 할 질문을 먼저 정리해줘.
2. 네가 코드에서 확인해야 할 것도 따로 정리해줘.
3. 그다음 작업을 실패 가능성이 낮은 작은 단계로 쪼개줘.
4. 각 단계마다 산출물과 검증 방법을 적어줘.
5. 아직 구현하지 마.
```

계획이 나온 뒤에는 이렇게 말하면 됩니다.

```text
계획의 1단계만 진행해줘.

이번 단계 밖의 작업은 하지 마.
구현 후에는 변경 파일, 산출물, 검증 결과를 알려줘.
진행하다가 단계가 너무 크다고 판단되면 더 작게 쪼개고 멈춰.
```

버그가 나면 이렇게 말하면 됩니다.

```text
지금 [문제 상황]이 발생한다.

바로 수정하지 말고 원인부터 특정해줘.
확인할 후보는 네가 현재 코드 구조를 보고 정해줘.

원인을 특정한 뒤 가장 작은 수정만 적용해줘.
수정 후 원인, 변경 파일, 검증 결과를 알려줘.
```

방향이 바뀌면 이렇게 말하면 됩니다.

```text
목표가 바뀌었다.

새 흐름은 이렇다.
[새로 원하는 사용자 흐름]

기존 구현을 억지로 유지하지 말고,
새 흐름 기준으로 유지할 것과 제거할 것을 먼저 나눠줘.
그다음 작은 단계로 다시 계획해줘.
아직 구현하지 마.
```

## Plan Mode는 언제 쓰면 좋을까?

Plan Mode는 쉽게 말해 **AI가 바로 파일을 고치지 못하게 막고, 먼저 읽고 계획하게 만드는 모드**입니다.

항상 켤 필요는 없습니다.
버튼 문구 수정, 간단한 CSS 조정, 이미 계획이 확정된 작은 단계라면 일반 모드로 바로 진행해도 됩니다.

Plan Mode가 특히 좋은 순간은 따로 있습니다.

1. 여러 파일이 함께 바뀔 것 같을 때
2. 데이터 구조, API, 저장 로직, 인증, 결제처럼 깨지면 큰 범위에 영향이 갈 때
3. 내가 원하는 사용자 흐름은 알지만 구현 방식은 잘 모를 때
4. 기존 구조를 먼저 읽어야 안전한 작업일 때
5. 중간에 방향이 바뀌어서 유지할 것과 제거할 것을 다시 나눠야 할 때
6. AI가 한 번에 너무 많이 고칠 것 같은 느낌이 들 때

각 단계를 진행할 때도 기준은 같습니다.
처음 큰 계획을 세울 때는 Plan Mode가 유리합니다.
하지만 계획이 이미 작게 쪼개졌고, 지금 단계가 명확하다면 굳이 매번 Plan Mode를 켤 필요는 없습니다.

대신 다음 상황에서는 다시 Plan Mode로 멈추는 편이 좋습니다.

1. 진행하다 보니 이번 단계가 생각보다 커졌을 때
2. 이전 단계 결과 때문에 다음 계획을 바꿔야 할 때
3. 저장, 삭제, 전송, 마이그레이션 같은 되돌리기 어려운 연결을 하기 직전일 때
4. 테스트가 실패했는데 원인이 명확하지 않을 때

요약하면 이렇습니다.

```text
작고 명확한 실행은 일반 모드.
크고 애매한 판단은 Plan Mode.
```

Plan Mode는 속도를 늦추는 장치가 아닙니다.
불필요한 수정을 막아서 전체 재작업 시간을 줄이는 장치입니다.

Skill은 Plan Mode와 조금 다릅니다.
Skill은 "코드 리뷰", "디자인 리뷰", "QA", "디버깅"처럼 반복해서 쓰는 작업 방식을 저장해둔 지침입니다.
매번 길게 설명하지 않고 특정 관점으로 검토하게 할 때 유용합니다.

하지만 초보자라면 일단 이것만 기억해도 충분합니다.

```text
바로 구현하지 말고 먼저 계획해줘.
작업을 작은 단계로 나눠줘.
각 단계마다 산출물과 검증 방법을 적어줘.
```

## TECHSPEC.md 같은 산출물은 꼭 필요할까?

항상 필요하지는 않습니다.

작은 수정이라면 AI가 마지막에 정리해주는 변경 파일, 바뀐 동작, 검증 결과만으로 충분합니다.
모든 작업마다 TECHSPEC.md를 만들면 문서가 너무 많아지고, 금방 낡습니다.

하지만 아래 같은 작업이라면 문서를 남기는 편이 좋습니다.

1. 하루 안에 끝나지 않을 만큼 기능이 큰 경우
2. 여러 화면, 여러 API, 여러 데이터 흐름이 엮이는 경우
3. 나중에 왜 이렇게 만들었는지 다시 봐야 할 결정이 있는 경우
4. 여러 사람이 보거나, 다른 AI 세션에 넘겨야 하는 경우
5. 마이그레이션, 권한, 결제, 저장 구조처럼 실수 비용이 큰 경우

이때 TECHSPEC.md는 멋진 설계 문서일 필요가 없습니다.
AI와 작업을 이어가기 위한 현재 기준표이면 충분합니다.

문서에는 보통 아래 정도만 있으면 됩니다.

1. 목표와 사용자 흐름
2. 이번 작업 범위와 하지 않을 일
3. 현재 결정된 구조
4. 단계별 계획과 진행 상태
5. 변경된 주요 파일
6. 검증한 내용
7. 남은 위험과 질문

문서 위치는 프로젝트 성격에 맞추면 됩니다.

1. 팀이나 프로젝트에 남길 문서라면 `docs/specs/<feature-name>.md`처럼 둔다
2. 기능 하나만 계속 추적한다면 `TECHSPEC.md`나 `FEATURE_SPEC.md`로 둔다
3. 임시 작업 메모라면 git에 올리지 않는 `.context/` 같은 작업용 폴더에 둔다
4. 작은 작업이라면 별도 파일 없이 마지막 요약만 받는다

가장 중요한 건 문서를 만든 뒤 방치하지 않는 것입니다.
코드와 맞지 않는 TECHSPEC.md는 없는 것보다 헷갈릴 수 있습니다.

문서를 남기는 작업이라면 각 단계가 끝날 때 이렇게 시키면 됩니다.

```text
이번 단계 결과를 TECHSPEC.md에 반영해줘.

새로 결정된 내용, 바뀐 계획, 검증 결과만 업데이트해줘.
코드와 맞지 않는 오래된 내용은 삭제하거나 outdated로 표시해줘.
문서만 그럴듯하게 늘리지 말고, 다음 작업자가 이어서 볼 정보만 남겨줘.
```

작업이 끝났을 때도 한 번 정리시키는 게 좋습니다.

```text
최종 구현 결과 기준으로 TECHSPEC.md를 정리해줘.

완료된 계획은 완료 상태로 표시하고,
실제로 구현되지 않은 아이디어는 제거하거나 남은 과제로 분리해줘.
검증 결과와 남은 위험만 마지막에 짧게 남겨줘.
```

문서 관리는 어렵게 생각할 필요가 없습니다.
**나중에 다시 봐야 하는 판단만 남기고, 일회성 대화는 요약으로 끝내면 됩니다.**

## 결론

AI 코딩을 잘한다는 건 프롬프트를 화려하게 쓰는 것이 아닙니다.

빠른 사람은 AI에게 이렇게 일을 시킵니다.

1. 내가 원하는 사용자 흐름을 이해해라
2. 부족한 요구사항은 질문해라
3. 현재 코드 구조를 읽어라
4. 작업을 작게 쪼개라
5. 각 단계의 산출물과 검증 방법을 정해라
6. 한 단계씩 구현하고 검증해라
7. 막히면 추측하지 말고 원인부터 찾아라
8. 방향이 바뀌면 기존 구현에 억지로 붙이지 말고 다시 정리해라
9. 크고 애매한 판단은 Plan Mode에서 먼저 멈춰라
10. 나중에 다시 볼 결정만 TECHSPEC.md 같은 문서로 남겨라

"그냥 짜 줘"는 빨라 보이지만 재작업을 부릅니다.
반대로 사용자가 모든 기술 계획을 직접 짜려고 해도 느려집니다.

가장 빠른 방식은 중간입니다.

**내 머릿속에 있는 기능을 AI와 함께 충분히 구체화하고, AI가 현실 코드로 옮길 수 있을 만큼 작업을 작게 쪼개서, 계획물·산출물·검증 결과를 계속 받는 것.**

그게 복잡한 일을 AI에게 제대로 맡기는 방법입니다.
