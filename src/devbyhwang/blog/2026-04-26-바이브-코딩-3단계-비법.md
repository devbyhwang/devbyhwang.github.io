"그냥 짜 줘"가 가장 빨라 보이지만, 실제로는 가장 느립니다.<br />
처음 출력은 빠른데, 곧바로 재작업이 시작됩니다.<br />
요구사항 다시 설명하고, 범위 줄이고, 깨진 부분 고치다 보면 결국 시간이 두 배, 세 배로 늘어납니다.<br />
<br />
속도를 진짜 올리려면 한 번에 크게 시키는 게 아니라, **AI가 딴길로 못 새게 작업 시스템을 먼저 깔아야 합니다.**<br />
제가 실무에서 계속 쓰는 방식은 딱 3단계입니다.<br />
중요한 건 사용자가 계획을 완벽하게 짜야 한다는 뜻이 아닙니다.<br />
계획을 AI에게 시키고, 그 계획을 작게 검산할 줄 알면 됩니다.<br />
<br />
## 1단계: 먼저 구현 금지, Plan-first로 고정<br />
처음 요청은 "코드 작성"이 아니라 "계획 작성"이어야 합니다.<br />
<br />
나쁜 요청:<br />

```text
리액트로 장바구니 기능 만들어줘.
```

좋은 요청:<br />

```text
바로 구현하지 말고, 먼저 단계별 구현 계획을 작성해줘.

목표:
사용자가 상품을 장바구니에 담고, 수량을 변경하고, 결제 전 총액을 확인하게 한다.

이번 범위:
- 장바구니 목록 표시
- 수량 증가/감소
- 총액 실시간 갱신

범위 제외:
- 결제 API 연동
- 쿠폰/배송비 계산

제약:
- 기존 상태 관리 구조 유지
- 장바구니 페이지 외 UI 변경 금지

검증:
- 수량 변경 시 총액이 즉시 반영되는지
- 기존 장바구니 삭제 기능이 깨지지 않는지

진행방식:
- 먼저 계획만 제출
- 내가 승인한 뒤에만 코드 수정
```

핵심은 간단합니다.<br />
**계획 승인 전에는 구현 못 하게 막는 것.**<br />
이 한 줄만 지켜도 불필요한 왕복이 크게 줄어듭니다.<br />
<br />
## 2단계: 기능을 "검산 가능한 최소 단위"로 쪼개기<br />
큰 기능을 한 번에 던지면, AI는 보통 과하게 구현합니다.<br />
그래서 "작업 단위"가 아니라 **검산 단위**로 잘라야 합니다.<br />
<br />
나쁜 요청:<br />

```text
회원가입 기능 만들어줘.
```

좋은 요청:<br />

```text
회원가입 기능을 아래 순서로 나눠 계획해줘.

1. 현재 auth/route 구조 확인
2. 회원가입 폼 UI 추가 (UI만)
3. 입력값 validation 추가
4. API 요청 함수 추가
5. 성공/실패 상태 처리
6. 로그인 화면 이동 처리
7. 테스트 및 회귀 확인

각 단계마다 아래를 반드시 포함해줘.
- 수정 파일
- 수정하면 안 되는 파일
- 검증 명령
```

그리고 단계 지시를 더 좁힙니다.<br />

```text
이번 단계는 UI만 만든다.
API 호출 금지.
전역 상태 저장 금지.
폼 필드/버튼/기본 에러 메시지만 구현한다.
```

이렇게 쪼개면 빨라지는 이유가 명확합니다.<br />
문제가 생겨도 원인 구간이 바로 특정되고, 롤백 범위도 작아집니다.<br />
<br />
## 3단계: 상황별 역할/스킬을 지정해서 일 시키기<br />
AI를 한 가지 모드로만 쓰면 효율이 급락합니다.<br />
상황마다 역할을 바꿔야 속도와 정확도가 같이 올라갑니다.<br />
<br />
기획이 흐릴 때:<br />

```text
구현 전에 제품 관점으로 먼저 정리해줘.
사용자 흐름, 필수 기능, 후순위 기능으로 분리해줘.
```

구조가 복잡할 때:<br />

```text
이 계획을 엔지니어링 리뷰해줘.
데이터 흐름, 상태 관리, edge case, 테스트 누락만 집중해서 봐줘.
```

버그가 났을 때:<br />

```text
바로 수정하지 말고 원인부터 특정해줘.
이벤트 바인딩, 상태 업데이트, 렌더 타이밍 중 어디서 끊기는지 먼저 확인해줘.
원인 특정 후 최소 수정안만 적용해줘.
```

UI가 어색할 때:<br />

```text
디자인 리뷰 모드로 점검해줘.
정보 위계, 간격, 클릭 영역, 모바일 깨짐만 체크하고 최소 수정안만 제시해줘.
```

기능 완료 확인할 때:<br />

```text
QA 관점으로 시나리오 테스트해줘.
클릭, 입력, 새로고침, 실패 상태, 모바일까지 확인하고
먼저 리포트만 작성해줘. 바로 수정은 하지 마.
```

한 줄 요약: **역할을 지정하지 않으면 AI가 알아서 판단하고, 그 판단이 재작업을 만듭니다.**<br />
<br />
## 중요한 보완: 계획을 잘 몰라도 괜찮다<br />
여기서 한 가지 오해가 생길 수 있습니다.<br />
"계획을 먼저 세워야 한다"는 말은 사용자가 처음부터 좋은 계획을 다 짜야 한다는 뜻이 아닙니다.<br />
오히려 실전에서는 반대에 가깝습니다.<br />
<br />
사용자는 보통 원하는 결과는 압니다.<br />
하지만 그 결과를 만들기 위해 어떤 파일을 고쳐야 하는지, 어떤 순서로 나눠야 하는지, 어떤 검증을 해야 하는지는 모릅니다.<br />
그걸 전부 사람이 직접 정리하려고 하면 AI를 쓰는 의미가 줄어듭니다.<br />
<br />
더 빠른 방식은 이겁니다.<br />
<br />
**내가 계획을 다 짜는 게 아니라, AI에게 계획을 짜게 만들고 그 계획을 단계별로 승인하는 것.**<br />
<br />
예를 들어 이렇게 시작하면 됩니다.<br />

```text
내가 만들고 싶은 건 대략 이런 흐름이야.

사용자가 그래프에서 포인트를 더블클릭하면,
그 포인트 안으로 들어가는 느낌이 나고,
왼쪽에는 현재 상황에 맞는 블록 후보가 나오고,
오른쪽에는 글을 쓰는 영역이 열렸으면 좋겠어.

나는 세부 구현 순서를 잘 모른다.
바로 구현하지 말고 현재 코드 구조를 먼저 확인한 뒤,
작업을 실패 가능성이 낮은 최소 단위로 쪼개서 계획해줘.

각 단계마다 아래를 포함해줘.
- 이 단계의 목표
- 수정할 가능성이 높은 파일
- 절대 건드리면 안 되는 범위
- 완료 기준
- 검증 방법
- 다음 단계로 넘어가기 전에 확인할 것
```

이 요청의 핵심은 "내가 모르니까 네가 계획해"입니다.<br />
AI에게 코드를 맡기기 전에, **작업을 어떻게 쪼갤지도 먼저 맡기는 것**입니다.<br />
<br />
그러면 사용자는 모든 기술 디테일을 알 필요가 없습니다.<br />
대신 아래 세 가지만 보면 됩니다.<br />
<br />
1. 내가 원하는 사용자 흐름과 맞는가
2. 이번 단계가 너무 크지 않은가
3. 검증 기준이 분명한가

계획이 너무 추상적이면 이렇게 다시 시키면 됩니다.<br />

```text
이 계획은 아직 너무 커.
실패해도 원인을 바로 찾을 수 있게 더 작은 단위로 쪼개줘.

각 단계는 한 번에 하나의 변화만 하게 해줘.
예:
- UI 뼈대만 추가
- 데이터 연결만 추가
- 클릭 이벤트만 연결
- 저장/복원만 추가
- 검증만 수행
```

반대로 AI가 너무 많이 하려고 하면 이렇게 막습니다.<br />

```text
이번 단계에서는 구현하지 말고 문서화만 해.
데이터 파일은 수정하지 마.
UI 파일도 수정하지 마.
후속 단계에서 실제 반영할 후보만 정리해.
```

또는 이렇게 범위를 고정합니다.<br />

```text
이번 단계는 실제 데이터 반영만 해.
UI 변경 금지.
추천 로직 변경 금지.
필드 이름 변경 금지.
검증까지 하고 멈춰.
```

이 방식이 좋은 이유는, 사용자가 처음부터 완벽한 설계자가 될 필요가 없기 때문입니다.<br />
사용자는 "원하는 방향"을 말하고, AI는 "실행 가능한 단계"로 바꿉니다.<br />
그리고 사용자는 각 단계가 맞는지만 확인합니다.<br />
<br />
실제로 작업하다 보면 아래 흐름이 가장 안정적입니다.<br />
<br />
1. 사용자가 큰 방향을 말한다
2. AI가 단계별 계획을 만든다
3. 사용자가 1~3단계만 진행하라고 한다
4. AI가 구현하거나 문서화한다
5. 검증 결과를 보고 다음 단계로 넘어간다
6. 중간에 방향이 바뀌면 계획을 다시 업데이트한다

중요한 건 한 번 만든 계획에 갇히지 않는 것입니다.<br />
AI 작업은 처음 계획보다, **계획을 계속 작게 고쳐가며 진행하는 방식**이 훨씬 강합니다.<br />
<br />
예를 들어 작업 중에 이런 일이 자주 생깁니다.<br />
<br />
"처음엔 구름 UI로 선택하려고 했는데, 써보니 우측 패널 목록이 낫겠다."<br />
"우측 패널도 복잡하니, 그냥 드래그 드롭 작성 캔버스로 줄이자."<br />
"기본 블록이 자동으로 들어오니 오히려 방해된다. 처음엔 비워두자."<br />
<br />
이때는 AI에게 이렇게 말합니다.<br />

```text
지금까지 한 작업 기준으로 목표가 바뀌었어.
기존 구현을 무리하게 유지하지 말고,
새 흐름에 맞게 다음 단계 계획을 다시 작성해줘.

현재 원하는 흐름:
1. 포인트 더블클릭
2. 왼쪽에 블록 후보 표시
3. 오른쪽은 빈 작성 캔버스
4. 드래그 드롭으로만 블록 생성
5. 블록 안에서 섹션을 추가하며 작성

이 기준으로 기존 UI 중 제거할 것과 유지할 것을 나눠줘.
```

그냥 "이거 바꿔줘"라고 하면 AI는 기존 구조 위에 덧붙이려는 경향이 있습니다.<br />
하지만 "새 흐름 기준으로 제거할 것과 유지할 것을 나눠줘"라고 하면, 불필요한 UI를 걷어내는 쪽으로 생각하게 됩니다.<br />
<br />
결국 AI와 일하는 속도는 프롬프트를 길게 쓰는 능력이 아닙니다.<br />
**내가 모르는 부분까지 AI에게 계획하게 만들고, 그 계획을 작게 검산하며 진행하는 능력**입니다.<br />
<br />
처음부터 세부 계획을 잘 짜지 못해도 됩니다.<br />
대신 이렇게 말하면 됩니다.<br />

```text
나는 최종 사용자 흐름만 알고 있고,
구현 순서와 세부 작업 단위는 잘 모른다.

네가 먼저 현재 코드 구조를 확인하고,
실패 가능성이 낮은 최소 단계들로 계획을 만들어줘.

각 단계는 내가 승인하면 진행하고,
진행 후에는 무엇이 바뀌었는지와 검증 결과를 알려줘.
```

이 한 문장이 있으면 사용자는 훨씬 적은 에너지로 AI를 통제할 수 있습니다.<br />
AI가 알아서 폭주하지 못하게 하면서도, 사용자가 모든 구현 디테일을 직접 설계하지 않아도 됩니다.<br />
<br />
## 실전 예시: 보드 카드 드래그앤드롭 구현<br />
위 원칙을 일반적인 기능 구현에 적용하면 이렇게 됩니다.<br />
아래 순서대로 복붙해서 쓰면 됩니다.<br />
<br />
첫 요청 (계획 전용):<br />

```text
프로젝트 보드에서 카드를 컬럼 간 드래그 이동하게 만들고 싶다.

바로 구현하지 말고 계획만 작성해줘.
현재 코드 구조를 먼저 확인하고, 가장 작은 단위로 쪼개서 작성해줘.

이번 범위:
- drag 시작
- 컬럼 drop
- card.columnId 변경
- 화면 재정렬

범위 제외:
- 서버 저장
- 권한 처리
- 멀티 선택
- 모바일 drag UX

제약:
- 기존 카드 렌더링 컴포넌트 구조 유지
- 스타일 전면 리팩터링 금지

검증:
- 같은 컬럼/다른 컬럼 drop 모두 동작
- drop 실패 시 원위치
- 기존 클릭/열기 동작 유지

진행방식:
- 계획 제출 → 승인 후 단계별 구현
```

1단계 실행 요청:<br />

```text
1단계만 진행해줘.
이번 단계는 drag 시작 상태만 추가한다.
drop 처리 금지.
데이터 변경 금지.
검증 방법까지 같이 제출해줘.
```

2단계 실행 요청:<br />

```text
2단계만 진행해줘.
컬럼 dragover/drop 이벤트만 연결한다.
아직 카드 위치는 바꾸지 마.
드롭 가능한 영역 표시까지만 구현해줘.
```

이 방식의 장점은 단순합니다.<br />
AI와 같은 지도를 보고, 한 칸씩 전진하게 만든다는 점입니다.<br />
<br />
## 속도 공식 (체크리스트)<br />
아래 4개가 매 요청에 들어가면 속도가 안정적으로 붙습니다.<br />
<br />
1. **계획**: 세부를 모르겠으면 AI에게 계획부터 만들게 하기
2. **분할**: 검산 가능한 최소 단위로 쪼개기
3. **역할**: 기획/엔지니어링/디버깅/디자인/QA 역할 지정
