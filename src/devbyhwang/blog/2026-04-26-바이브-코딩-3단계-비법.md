"그냥 짜 줘"가 가장 빨라 보입니다.<br />
하지만 실제로는 자주 느립니다.<br />
처음 코드는 빨리 나오지만, 곧바로 이런 일이 생깁니다.<br />
<br />
"왜 기존 기능이 깨졌지?"<br />
"이건 내가 원한 흐름이 아닌데?"<br />
"수정했더니 다른 화면이 이상해졌네?"<br />
<br />
문제는 AI가 느린 게 아닙니다.<br />
**AI에게 일을 너무 크게, 너무 흐릿하게 맡긴 것**이 문제입니다.<br />
<br />
그렇다고 사용자가 구현 계획을 직접 다 짜야 한다는 뜻은 아닙니다.<br />
오히려 반대입니다.<br />
사용자가 모든 파일 구조, 상태 흐름, 테스트 전략을 직접 정리하려고 하면 AI를 쓰는 의미가 줄어듭니다.<br />
<br />
더 빠른 방식은 이겁니다.<br />
<br />
**복잡한 분석과 계획은 AI에게 시키고, 사용자는 원하는 결과와 승인 기준만 잡는다.**<br />
<br />
제가 실무에서 쓰는 방식은 크게 3단계입니다.<br />
<br />
1. AI에게 먼저 현재 구조를 읽게 한다
2. AI에게 실패 가능성이 낮은 최소 단위 계획을 만들게 한다
3. AI에게 구현 후 검증까지 하게 하고, 막히면 원인부터 찾게 한다

여기에 작업이 커지면 Plan Mode나 skill을 붙입니다.<br />
Plan Mode는 AI가 바로 코드를 고치지 못하게 멈춰 세우는 실행 모드이고, skill은 디자인·엔지니어링·제품 관점의 검토 방식을 저장해둔 지침입니다.<br />
<br />
핵심은 사용자가 똑똑하게 모든 걸 직접 설계하는 게 아닙니다.<br />
**AI가 알아서 폭주하지 못하게 작업 방식을 시키는 것**입니다.<br />
<br />
## 1단계: "구현해줘" 전에 "먼저 읽고 계획해줘"<br />
가장 흔한 실수는 바로 구현부터 시키는 것입니다.<br />
예를 들어 이런 요청입니다.<br />
<br />
나쁜 요청:<br />

```text
그래프에서 포인트를 더블클릭하면 안으로 들어가고,
왼쪽에는 블록 후보,
오른쪽에는 글쓰기 영역이 열리게 만들어줘.
```

이렇게 요청하면 AI는 보통 바로 코드를 씁니다.<br />
문제는 현재 코드 구조를 충분히 확인하지 않은 채, 새 이벤트나 새 화면 구조를 얹을 수 있다는 점입니다.<br />
처음에는 그럴듯하지만 나중에 기존 클릭, 드래그, 줌 동작이나 저장 흐름과 충돌할 수 있습니다.<br />
<br />
더 좋은 요청은 이렇습니다.<br />

```text
내가 만들고 싶은 건 대략 이런 흐름이야.

사용자가 그래프에서 포인트를 더블클릭하면,
그 포인트 안으로 들어가는 느낌이 나고,
왼쪽에는 현재 상황에 맞는 블록 후보가 나오고,
오른쪽에는 글을 쓰는 영역이 열렸으면 좋겠어.

나는 세부 구현 순서를 잘 몰라.
바로 구현하지 말고 현재 코드 구조를 먼저 확인한 뒤,
작업을 실패 가능성이 낮은 최소 단위로 쪼개서 계획해줘.
```

이 요청에서 중요한 점은 사용자가 파일명, 상태 구조, 이벤트 처리 방식을 직접 정하지 않는다는 것입니다.<br />
사용자는 원하는 흐름만 말하고, AI에게 먼저 현재 구조를 읽게 합니다.<br />
그다음 AI가 안전한 구현 순서를 제안하게 만드는 것입니다.<br />
<br />
사용자는 모르는 것을 세부 체크리스트로 바꿔 적을 필요가 없습니다.<br />
오히려 AI에게 이렇게 말해야 합니다.<br />
<br />
**"나는 원하는 사용자 흐름은 안다. 구현 순서는 네가 구조를 읽고 계획해라."**<br />
<br />
그러면 AI는 바로 코드를 쓰기보다 현재 코드베이스를 읽고, 어디를 고쳐야 하는지부터 정리합니다.<br />
이 단계만 거쳐도 이상한 파일을 새로 만들거나 기존 구조와 어긋나는 일이 크게 줄어듭니다.<br />
<br />
## 2단계: 계획도 AI에게 더 작게 쪼개게 한다<br />
AI가 만든 계획이 처음부터 완벽할 필요는 없습니다.<br />
보통 첫 계획은 조금 큽니다.<br />
그러면 사용자가 직접 다시 설계하려고 하지 말고, AI에게 다시 쪼개라고 시키면 됩니다.<br />
<br />
예를 들어 AI가 이런 계획을 냈다고 가정해봅니다.<br />
<br />
```text
1. 그래프 더블클릭 추가
2. 안으로 들어가는 전환 추가
3. 좌우 패널 추가
4. 테스트
```

나쁘지는 않지만 아직 큽니다.<br />
이럴 때는 이렇게 다시 시킵니다.<br />
<br />
```text
이 계획은 아직 크다.
실패했을 때 원인을 바로 찾을 수 있게 더 작은 단위로 쪼개줘.

한 단계에 하나의 변화만 들어가게 해줘.
구체적인 단계와 기준은 네가 다시 채워줘.
아직 구현하지 마.
```

여기서 사용자가 하는 일은 계획을 직접 완성하는 게 아닙니다.<br />
AI가 낸 계획이 너무 크면, **"더 작게 쪼개"라고 다시 시키는 것**입니다.<br />
구체적인 단위 예시는 사용자가 미리 적을 필요가 없습니다.<br />
AI가 코드 구조를 본 뒤 "구조 확인 → 더블클릭 감지 → 선택 상태 저장 → 전환 효과 → 패널 뼈대 → 글쓰기 연결 → 최종 검증"처럼 다시 나누면 됩니다.<br />
<br />
이 방식이 빠른 이유는 단순합니다.<br />
한 번에 큰 기능을 만들면 버그가 났을 때 원인을 찾기 어렵습니다.<br />
작게 나누면 어느 단계에서 망가졌는지 바로 보입니다.<br />
<br />
## 3단계: 구현도 "한 번에 다"가 아니라 "검증까지 한 단계"로 시킨다<br />
계획이 잡혔다면 이제 구현을 시킵니다.<br />
하지만 여기서도 "전부 진행해줘"가 아니라, 검증 가능한 단위로 진행시키는 게 좋습니다.<br />
<br />
예를 들어 그래프 기능의 1단계라면 이렇게 요청합니다.<br />
<br />
```text
계획의 1단계만 진행해줘.

네가 작성한 1단계 계획대로 진행해줘.
계획 밖의 작업은 하지 마.
검증까지 끝낸 뒤 결과를 알려줘.
```

AI가 이 단계를 끝내면 다음에는 이렇게 이어갑니다.<br />
<br />
```text
2단계 진행해줘.

이전 단계에서 만든 구조를 기준으로 진행해줘.
이번 단계 밖의 작업은 하지 마.
검증까지 끝낸 뒤 결과를 알려줘.
```

이렇게 하면 사용자는 개발자가 아니어도 흐름을 통제할 수 있습니다.<br />
기술 디테일을 몰라도 됩니다.<br />
대신 아래 세 가지만 확인하면 됩니다.<br />
<br />
1. 이번 단계가 내가 원하는 결과와 맞는가
2. 이번 단계가 너무 크지 않은가
3. AI가 검증 결과를 보고했는가

## 막혔을 때는 "고쳐줘"보다 "원인부터 찾아줘"<br />
AI 작업에서 가장 시간이 많이 날아가는 순간은 버그가 났을 때입니다.<br />
이때 바로 "고쳐줘"라고 하면 AI가 추측으로 수정할 수 있습니다.<br />
그러면 다른 곳이 깨질 수 있습니다.<br />
<br />
버그가 났을 때는 이렇게 시키는 편이 훨씬 빠릅니다.<br />
<br />
```text
지금 저장 버튼을 눌러도 메모가 추가되지 않는다.

바로 수정하지 말고 원인부터 특정해줘.
확인할 지점은 네가 코드 구조를 보고 정해줘.
원인을 찾은 뒤 가장 작은 수정만 적용해줘.
수정 후에는 어떤 원인이었는지와 검증 결과를 알려줘.
```

이 요청의 핵심은 **수정 전에 원인을 찾게 하는 것**입니다.<br />
AI는 코드를 빨리 바꿀 수 있지만, 빨리 바꾸는 것이 항상 빠른 길은 아닙니다.<br />
원인을 모른 채 수정하면 같은 문제가 다시 생깁니다.<br />
<br />
특히 프론트엔드에서는 아래 지점 중 하나에서 자주 끊깁니다.<br />
<br />
1. 이벤트가 안 걸림
2. 상태는 바뀌는데 화면이 안 바뀜
3. 화면은 바뀌는데 저장이 안 됨
4. 저장은 되는데 복원이 안 됨
5. 기존 UI 위에 새 UI가 겹침

사용자가 이 순서를 직접 다 적을 필요는 없습니다.<br />
"이벤트, 상태, 렌더링, 저장 흐름을 보고 원인을 찾아줘" 정도만 말해도 AI가 확인 순서를 잡을 수 있습니다.<br />
<br />
## 방향이 바뀌면 "기존 구현을 유지하지 말고 다시 정리해줘"<br />
실제 작업을 하다 보면 처음 생각이 바뀝니다.<br />
이건 실패가 아닙니다.<br />
오히려 AI로 만들 때는 중간에 방향을 바꾸는 비용이 낮습니다.<br />
<br />
중요한 건 AI가 기존 구현 위에 계속 덧붙이지 않게 하는 것입니다.<br />
방향이 바뀌면 이렇게 말해야 합니다.<br />
<br />
```text
지금까지 구현한 것을 기준으로 목표가 바뀌었다.

기존 구현을 억지로 유지하지 말고,
새 사용자 흐름에 맞게 유지할 것과 제거할 것을 먼저 나눠줘.

새 흐름:
1. 처음에는 그래프 안에 구름처럼 떠 있는 UI에서 블록을 선택하려고 했다.
2. 그런데 직접 써보니 화면이 복잡하고, 어떤 후보를 골라야 하는지 한눈에 보기 어렵다.
3. 블록 후보는 그래프 안에 띄우지 말고 우측 패널 목록으로 보여주는 편이 낫겠다.
4. 그래프는 현재 위치와 맥락을 보여주는 역할만 하고, 선택과 편집은 우측 패널에서 하게 바꾸고 싶다.

먼저 계획만 작성해줘.
구현은 아직 하지 마.
```

그다음 이렇게 실행합니다.<br />
<br />
```text
계획의 1단계만 진행해줘.

새로 작성한 계획의 1단계대로 진행해줘.
기존 추천 알고리즘, 저장 로직, 그래프 기본 동작은 건드리지 마.
검증까지 하고 결과를 알려줘.
```

이렇게 말하면 AI가 "기존에 만든 게 아까우니 계속 살리자"는 식으로 가지 않습니다.<br />
새 목표 기준으로 걷어낼 것을 걷어내게 됩니다.<br />
처음 아이디어가 틀렸다는 뜻이 아닙니다.<br />
구름 UI는 상상할 때는 좋아 보일 수 있지만, 실제로 써보면 목록이 더 빠르고 명확할 수 있습니다.<br />
그럴 때 중요한 건 "구름 UI를 어떻게든 개선해줘"가 아니라, **새로 배운 것을 기준으로 구조를 다시 정리해달라고 말하는 것**입니다.<br />
<br />
## 실전 예시: 예약 가능 시간 추천 기능 만들기<br />
이번에는 다른 예시를 들어보겠습니다.<br />
팀 캘린더 앱에 "예약 가능한 시간 추천" 기능을 넣고 싶다고 가정합니다.<br />
사용자는 정확한 구현 방식을 모릅니다.<br />
그래도 괜찮습니다.<br />
AI에게 이렇게 시키면 됩니다.<br />
<br />
첫 요청:<br />

```text
팀 캘린더에서 참석자들의 빈 시간을 보고 회의 가능 시간을 추천하는 기능을 만들고 싶다.

내가 원하는 사용자 흐름:
1. 참석자를 고른다.
2. 가능한 회의 시간 후보가 몇 개 나온다.
3. 후보를 누르면 회의 초안에 시간이 들어간다.

나는 구현 순서를 잘 모른다.
바로 구현하지 말고 현재 코드 구조를 먼저 확인해줘.

기술적으로 무엇을 확인해야 하는지는 네가 정해줘.
코드 구조를 본 뒤 가장 안전한 구현 순서를 짧은 단계로 제안해줘.
각 단계는 내가 "진행" 또는 "보류"라고 답할 수 있을 만큼 작게 나눠줘.

아직 구현하지 마.
```

여기서 사용자는 캘린더 이벤트 데이터 구조나 시간대 처리 방식을 직접 알 필요가 없습니다.<br />
그런 항목은 AI가 코드 구조를 보고 확인해야 합니다.<br />
사용자가 해야 할 일은 "회의 가능 시간을 추천받고 싶다"는 사용자 흐름을 말하는 것입니다.<br />
<br />
AI가 계획을 내면, 사용자는 세부 구현 항목을 다시 쓰지 말고 크기만 판단하면 됩니다.<br />
<br />
```text
계획이 아직 크다.
실패했을 때 원인을 바로 찾을 수 있게 더 작게 쪼개줘.

각 단계는 한 번에 하나의 변화만 하게 해줘.
저장이나 전송처럼 되돌리기 어려운 연결은 뒤로 미뤄줘.
세부 기준은 네가 다시 채워줘.
```

AI가 예를 들어 이렇게 쪼갔다고 가정해봅니다.<br />
<br />
```text
1. 현재 선택된 날짜와 참석자 이벤트를 읽을 수 있는지 확인
2. 읽어온 이벤트로 빈 시간 후보만 계산
3. 후보를 추천 패널에 표시
4. 후보 클릭 시 회의 초안 시간에 반영
5. 저장/전송 API 연결
6. 실패 상태와 빈 상태 처리
```

이제 사용자는 각 단계의 세부 검증 기준을 새로 쓰지 않아도 됩니다.<br />
AI가 만든 계획을 기준으로 이렇게만 이어가면 됩니다.<br />
<br />
1단계 실행:<br />

```text
계획의 1단계만 진행해줘.

네가 작성한 1단계 계획대로 진행해줘.
이번 단계 밖의 작업은 하지 마.

검증까지 끝낸 뒤 변경 파일과 결과를 알려줘.
```

2단계 실행:<br />

```text
2단계 진행해줘.

이전 단계에서 확인한 내용을 기준으로 진행해줘.
이번 단계 밖의 작업은 하지 마.
네가 작성한 2단계 계획대로 진행해줘.
```

3단계 실행:<br />

```text
3단계 진행해줘.

네가 작성한 3단계 계획대로 진행해줘.
이번 단계 밖의 작업은 하지 마.
검증까지 끝낸 뒤 다음 단계에서 확인해야 할 것을 알려줘.
```

이 흐름에서 사용자는 알고리즘을 직접 설계하지 않았습니다.<br />
파일 구조도 직접 찾지 않았습니다.<br />
테스트 케이스도 처음부터 다 알 필요가 없었습니다.<br />
<br />
대신 AI에게 계속 이렇게 시켰습니다.<br />
<br />
1. 먼저 읽어라
2. 작게 나눠라
3. 이번 단계만 해라
4. 하지 말아야 할 일을 지켜라
5. 검증하고 보고해라

이게 빠릅니다.<br />
<br />
## AI에게 맡길수록 사용자는 무엇을 말해야 하나<br />
AI에게 많이 맡긴다는 건 아무렇게나 맡긴다는 뜻이 아닙니다.<br />
다만 사용자가 명확하게 말해야 하는 것은 기술 체크리스트가 아닙니다.<br />
사용자가 말해야 하는 것은 판단 기준입니다.<br />
<br />
1. 최종 사용자는 무엇을 하려는가
2. 이번에 꼭 포함되어야 하는 흐름은 무엇인가
3. 이번에는 미뤄도 되는 것은 무엇인가
4. 절대 깨지면 안 되는 기존 동작은 무엇인가
5. 막히면 추측해서 고칠지, 질문하고 멈출지

반대로 AI에게 맡겨도 되는 것은 이런 것들입니다.<br />
<br />
1. 어떤 파일을 읽어야 하는지
2. 어떤 데이터 흐름을 확인해야 하는지
3. 어떤 테스트나 검증이 필요한지
4. 어떤 단계로 나누는 게 안전한지
5. 이번 단계의 기술적인 금지 범위가 무엇인지
<br />
여기서 제일 중요한 건 "사용자가 직접 다 모르면 AI가 먼저 제안하게 한다"는 점입니다.<br />
**AI는 시키지 않으면 불필요한 곳까지 고칠 수 있지만, 사용자가 모든 금지 범위를 미리 알 수도 없습니다.**<br />
그래서 이렇게 말하는 편이 좋습니다.<br />
<br />
```text
내가 기술적인 금지 범위를 전부 알지는 못한다.

현재 코드 구조를 보고,
이번 단계에서 건드리면 위험한 범위를 네가 먼저 제안해줘.
영향이 큰 변경은 따로 표시하고,
이번 단계에서 피해야 할 작업을 계획에 포함해줘.
```

AI가 느려지는 이유는 코드를 못 짜서가 아닙니다.<br />
불필요한 변경을 하고, 그걸 다시 되돌리느라 느려지는 경우가 많습니다.<br />
금지 범위를 사용자가 혼자 다 쓰려고 하지 말고, AI가 먼저 초안을 만들게 하면 이 비용이 줄어듭니다.<br />
<br />
## Plan Mode와 Skill은 실제로 어떻게 쓰나<br />
여기서 헷갈리면 안 되는 점이 있습니다.<br />
Plan Mode는 그냥 "계획해줘"라고 말하는 프롬프트가 아닙니다.<br />
Claude Code 같은 도구에서는 실제로 켜고 끄는 모드입니다.<br />
<br />
Claude Code 기준으로는 이렇게 씁니다.<br />
<br />
1. CLI에서는 `Shift+Tab`을 눌러 모드를 바꾼다
2. 한 번의 요청만 계획 모드로 보내고 싶으면 `/plan [할 일]`처럼 입력한다
3. 처음부터 계획 모드로 시작하려면 `claude --permission-mode plan`으로 실행한다
4. VS Code, Desktop, claude.ai Code에서는 입력창 근처의 모드 선택 UI에서 Plan Mode를 고른다

Plan Mode에서는 AI가 코드를 읽고, 필요한 명령어로 구조를 조사하고, 계획을 제안할 수 있습니다.<br />
하지만 소스 파일을 바로 수정하지 않습니다.<br />
계획이 나오면 사용자는 승인하거나, 더 다듬으라고 하거나, 계속 계획을 수정하게 할 수 있습니다.<br />
<br />
그래서 작은 수정에는 굳이 Plan Mode를 켤 필요가 없습니다.<br />
버튼 문구 하나를 바꾸거나 CSS 간격을 조금 조정하는 정도라면 그냥 요청해도 됩니다.<br />
반대로 아래처럼 여러 요소가 엮이면 Plan Mode가 유리합니다.<br />
<br />
```text
그래프 포인트 더블클릭,
포인트 안으로 들어가는 전환,
왼쪽 블록 후보 패널,
오른쪽 글쓰기 영역,
선택 상태 저장,
기존 줌/드래그 동작 유지
```

이 정도가 되면 바로 구현하면 위험합니다.<br />
이때는 먼저 Plan Mode를 켜고, 짧게 이렇게만 적으면 됩니다.<br />
<br />
```text
내가 원하는 흐름은 이거야.
[사용자 흐름]

현재 코드 구조를 먼저 읽고,
안전한 구현 순서를 짧은 단계로 제안해줘.
아직 구현하지 마.
```

Skill은 Plan Mode와 다릅니다.<br />
Plan Mode가 "지금은 수정하지 말고 계획만 하라"는 실행 모드라면, Skill은 반복해서 쓰는 작업 방식을 저장해둔 지침입니다.<br />
Claude Code의 skill은 보통 `SKILL.md` 파일에 설명과 지침을 적어두고, 필요할 때 자동으로 불러오거나 `/skill-name`처럼 직접 실행할 수 있습니다.<br />
개인용 skill은 `~/.claude/skills/<skill-name>/SKILL.md`, 프로젝트용 skill은 `.claude/skills/<skill-name>/SKILL.md`에 둘 수 있습니다.<br />
<br />
Claude 쪽은 `SKILL.md` 파일로 이런 지침을 만들 수 있고, Codex 쪽도 앱과 워크플로에서 skills를 지원합니다.<br />
도구마다 설치 방식과 이름은 다르지만, 실제로 자주 쓸 만한 skill 유형은 비슷합니다.<br />
<br />
```text
code-review:
PR이나 큰 diff를 병합하기 전에 버그, 회귀, 빠진 테스트를 점검할 때

test-writer:
구현은 끝났지만 핵심 흐름, edge case, 회귀 테스트가 부족할 때

debug-investigate:
에러, 500, 깨진 버튼, 저장 실패처럼 원인부터 찾아야 할 때

qa:
사용자 흐름을 직접 실행해보고 깨지는 부분을 찾을 때

security-review:
권한, 입력 검증, 비밀값 노출, 의존성 위험을 확인할 때

design-review:
화면 배치, 인터랙션, 시각적 위계, 모바일 대응을 반복해서 점검할 때

eng-review:
상태 구조, 데이터 흐름, API 연결, 테스트 전략을 반복해서 점검할 때

performance-check:
페이지 로딩, 번들 크기, 느린 렌더링, 불필요한 요청을 점검할 때

docs-update:
기능을 바꾼 뒤 README, changelog, 사용법 문서를 맞출 때

release-check:
배포 전 문서, 변경 내역, 테스트, 위험 요소를 같은 방식으로 확인할 때
```

처음부터 전부 만들 필요는 없습니다.<br />
저라면 `code-review`, `debug-investigate`, `test-writer`, `qa`, `docs-update` 정도만 먼저 둡니다.<br />
그다음 디자인 작업이 많으면 `design-review`, 배포가 잦으면 `release-check`, 보안이 중요한 제품이면 `security-review`를 추가합니다.<br />

예를 들어 그래프 안으로 들어가서 글을 쓰는 기능은 단순한 UI 추가가 아닙니다.<br />
상태 흐름도 있고, 인터랙션도 있고, 사용자가 길을 잃지 않게 하는 제품 판단도 있습니다.<br />
그럴 때는 먼저 Plan Mode를 켠 뒤, 설치된 skill이 있다면 이런 식으로 요청합니다.<br />
<br />
```text
design-review 관점으로 먼저 봐줘.

사용자가 그래프 포인트를 더블클릭하면
그 포인트 안으로 들어가는 느낌이 나고,
왼쪽에는 블록 후보,
오른쪽에는 글쓰기 영역이 열리는 흐름을 생각 중이야.

헷갈릴 지점과 화면 배치 위험을 먼저 짚고,
그다음 안전한 구현 순서를 제안해줘.
아직 구현하지 마.
```

핵심은 단순합니다.<br />
**Plan Mode는 수정 전에 멈추는 장치이고, Skill은 특정 관점으로 반복 검토하게 만드는 장치입니다.**<br />
수정 범위가 넓거나, 사용자 흐름이 복잡하거나, 실패했을 때 어디서 망가졌는지 찾기 어려울 것 같으면 먼저 Plan Mode를 켭니다.<br />
그 위에 디자인·엔지니어링·QA처럼 특정 관점이 필요하면 skill을 붙이면 됩니다.<br />
<br />
## 바로 복사해서 쓰는 짧은 요청<br />
여기서 한 가지 더 중요합니다.<br />
템플릿이 너무 자세하면 사용자가 다시 일을 하게 됩니다.<br />
"수정할 파일", "절대 건드리면 안 되는 범위", "완료 기준", "검증 방법" 같은 항목을 프롬프트에 길게 쓰기 시작하면 다시 느려집니다.<br />
<br />
그래서 처음 요청은 길게 쓰지 않아도 됩니다.<br />
사용자는 대략적인 흐름만 말하고, 세부 계획은 AI가 현재 코드 구조를 읽은 뒤 채우게 하면 됩니다.<br />
실제로 사용자가 처음에 직접 적을 것은 세 가지면 충분합니다.<br />
<br />
1. 사용자가 어떤 흐름을 경험했으면 하는지
2. 아직 구현하지 말고 먼저 계획해야 한다는 것
3. 세부 계획 항목은 AI가 코드 구조를 보고 채워야 한다는 것
<br />
가장 빠른 30초 버전은 이렇습니다.<br />
<br />
```text
내가 만들고 싶은 건 대략 이런 흐름이야.

[사용자가 실제로 경험하길 원하는 흐름을 말하듯 적기]

나는 세부 구현 순서를 잘 모른다.
바로 구현하지 말고 현재 코드 구조를 먼저 확인한 뒤,
작업을 실패 가능성이 낮은 최소 단위로 쪼개서 계획해줘.

필요한 확인 항목과 세부 계획은 네가 채워줘.
내가 바로 승인하거나 거절할 수 있게 짧은 단계로 정리해줘.
확실하지 않은 것만 마지막에 질문해줘.

아직 구현하지 마.
```

예를 들어 이런 식입니다.<br />
<br />
```text
내가 만들고 싶은 건 대략 이런 흐름이야.

사용자가 그래프에서 포인트를 더블클릭하면,
그 포인트 안으로 들어가는 느낌이 나고,
왼쪽에는 현재 상황에 맞는 블록 후보가 나오고,
오른쪽에는 글을 쓰는 영역이 열렸으면 좋겠어.

나는 세부 구현 순서를 잘 모른다.
바로 구현하지 말고 현재 코드 구조를 먼저 확인한 뒤,
작업을 실패 가능성이 낮은 최소 단위로 쪼개서 계획해줘.

필요한 확인 항목과 세부 계획은 네가 채워줘.
내가 바로 승인하거나 거절할 수 있게 짧은 단계로 정리해줘.
확실하지 않은 것만 마지막에 질문해줘.

아직 구현하지 마.
```

이 요청에서 사용자가 직접 쓴 것은 사실상 사용자 경험뿐입니다.<br />
더블클릭, 안으로 들어가는 느낌, 왼쪽 블록 후보, 오른쪽 글쓰기 영역.<br />
그 정도면 충분합니다.<br />
<br />
나머지는 AI가 코드 구조를 보고 역산해야 합니다.<br />
예를 들어 그래프 렌더링 파일, 포인트 클릭 이벤트, 현재 선택 상태, 좌우 패널 레이아웃, 글쓰기 상태 저장 위치, 기존 블록 추천 로직이 있는지 등을 AI가 찾아야 합니다.<br />
사용자가 처음부터 그 파일명을 알아낼 필요는 없습니다.<br />
<br />
AI가 좋은 계획을 냈다면 대략 이런 모양이어야 합니다.<br />
아래처럼 자세한 항목은 사용자가 프롬프트에 직접 쓰는 것이 아니라, AI가 계획 결과로 채워줘야 하는 내용입니다.<br />
<br />
```text
1단계: 그래프 포인트 더블클릭 이벤트가 어디서 처리되는지 확인
- 이 단계의 목표: 기존 그래프 이벤트 구조 파악
- 수정할 가능성이 높은 파일: GraphView, NodeCanvas, 관련 hook
- 절대 건드리면 안 되는 범위: 데이터 저장 구조, 블록 생성 로직
- 완료 기준: 더블클릭을 붙일 위치가 명확해짐
- 검증 방법: 기존 클릭/드래그/줌 동작이 어디서 처리되는지 확인
- 다음 단계 전 확인할 것: 더블클릭이 기존 단일 클릭 동작과 충돌하지 않는지

2단계: 더블클릭 시 선택된 포인트 상태만 저장
- 이 단계의 목표: 화면 전환 없이 선택 상태만 추가
- 수정할 가능성이 높은 파일: graph state hook, selected point state
- 절대 건드리면 안 되는 범위: 패널 UI, 저장 API, 추천 로직
- 완료 기준: 더블클릭한 포인트 id가 상태에 들어감
- 검증 방법: 로그나 devtools로 선택 상태 확인
- 다음 단계 전 확인할 것: 다른 포인트를 더블클릭하면 상태가 바뀌는지

3단계: 포인트 안으로 들어가는 전환 느낌만 추가
- 이 단계의 목표: 실제 기능 연결 전에 시각 전환만 검증
- 수정할 가능성이 높은 파일: graph viewport, animation layer
- 절대 건드리면 안 되는 범위: 데이터 모델, 글쓰기 영역, 블록 후보
- 완료 기준: 더블클릭 시 해당 포인트 중심으로 확대/전환됨
- 검증 방법: 기존 zoom/pan 동작 유지 확인
- 다음 단계 전 확인할 것: 사용자가 다시 나올 방법이 필요한지
```

이렇게 AI가 초안을 만들어주면 사용자는 훨씬 빨리 판단할 수 있습니다.<br />
"맞다", "2단계가 너무 크다", "글쓰기 저장은 아직 빼자", "왼쪽 후보는 mock 데이터로 먼저 보자" 정도만 말하면 됩니다.<br />
<br />
구현 단계에서도 사용자가 모든 항목을 다시 채울 필요는 없습니다.<br />
이미 AI가 만든 계획을 기준으로 이렇게 짧게 지시하면 됩니다.<br />
<br />
```text
방금 계획 중 1단계만 진행해줘.

네가 작성한 1단계 계획대로 진행해줘.
진행하다가 1단계가 생각보다 크다고 판단되면 더 작게 쪼개고 멈춰.

구현 후에는 변경 파일, 바뀐 동작, 검증 결과를 짧게 알려줘.
```

다음 단계도 마찬가지입니다.<br />
<br />
```text
2단계 진행해줘.

이전 단계에서 확인한 구조를 기준으로 진행해줘.
이번 단계 밖의 작업은 하지 마.
검증까지 끝낸 뒤 결과를 알려줘.
```

버그가 나면 이렇게 씁니다.<br />
<br />
```text
지금 [문제 상황]이 발생한다.

바로 수정하지 말고 원인부터 특정해줘.

확인할 후보는 네가 현재 코드 구조를 보고 정해줘.
특히 이벤트, 상태 변경, 렌더링, 저장/복원, CSS 레이어 충돌을 확인해줘.

원인을 특정한 뒤 가장 작은 수정만 적용해줘.
수정 후 검증 결과를 알려줘.
```

## 결론: 빠른 사람은 계획을 직접 다 짜는 사람이 아니다<br />
AI 코딩에서 빠른 사람은 모든 기술 계획을 직접 짜는 사람이 아닙니다.<br />
빠른 사람은 AI에게 이렇게 일을 시킵니다.<br />
<br />
1. 현재 구조를 읽어라
2. 네가 계획을 짜라
3. 더 작게 쪼개라
4. 이번 단계만 해라
5. 검증하고 보고해라
6. 막히면 원인부터 찾아라
7. 방향이 바뀌면 유지할 것과 제거할 것을 다시 나눠라
8. 큰 변경 전에는 Plan Mode나 skill로 먼저 검토시켜라

"그냥 짜 줘"는 빨라 보이지만, 재작업을 부릅니다.<br />
반대로 사용자가 모든 항목을 직접 채우려고 해도 느려집니다.<br />
AI에게 복잡한 일을 맡기되, **사용자 흐름과 작업 방식만 분명히 주고 세부 계획은 AI가 채우게 하는 것**이 진짜 빠른 방법입니다.<br />
